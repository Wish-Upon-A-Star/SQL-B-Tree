# CmdProcessor 코드 스터디 피그마 채점 기준 v2

이 기준은 발표용 보드가 아니라, **코드 공부용 보드**를 채점하기 위한 것이다.

핵심 질문은 하나다.

> 이 보드만 보고도, IDE를 열었을 때 어디부터 읽어야 하는지 알고 실제 코드 흐름을 따라갈 수 있는가?

즉, 예쁘게 요약한 아키텍처 자료는 높은 점수를 받을 수 없다.  
실제 파일, 실제 함수, 실제 자료구조, 실제 읽는 순서, 실제 경계가 없으면 낮게 나온다.

## 1. 총점 구조

총점은 100점 만점이다.

- 학습 시작 가능성: 15점
- 실제 코드 추적 가능성: 20점
- 요청 생명주기 설명력: 15점
- 내부 자료구조 이해도: 15점
- 팀 경계 / 수정 경계 명확성: 10점
- 코드 조각 품질: 10점
- 스터디 순서 / 단계 설계: 10점
- 레이아웃 / 가독성: 5점

## 2. 필수 조건

아래 조건 중 하나라도 빠지면 높은 점수를 줄 수 없다.

- 실제 파일 경로가 4개 이상 보일 것
- 실제 함수명 또는 타입명이 8개 이상 보일 것
- 코드 조각이 4개 이상 있을 것
- 읽는 순서가 4단계 이상 보일 것
- “우리 역할”과 “다른 팀 역할”이 파일 수준에서 분리되어 있을 것
- request -> submit -> runtime -> engine 중 최소 3단계가 드러날 것

## 3. 항목별 기준

### A. 학습 시작 가능성 — 15점

- 0점: 이 보드를 보고 무엇부터 읽어야 할지 모르겠다.
- 5점: 대략적인 설명은 있으나 실제 첫 파일을 못 고르겠다.
- 10점: 첫 파일과 다음 파일 정도는 고를 수 있다.
- 15점: 보드만 보고도 “지금 어디부터 코드 읽기를 시작할지” 바로 결정할 수 있다.

### B. 실제 코드 추적 가능성 — 20점

- 0점: 코드와 무관한 설명 수준이다.
- 8점: 함수/파일 이름이 조금 보이지만 실제 코드 추적에는 부족하다.
- 14점: 실제 함수명과 파일 경계가 꽤 드러난다.
- 18점: submit에서 runtime까지 역추적이 가능하다.
- 20점: 보드를 옆에 두고 실제 코드 흐름을 무리 없이 따라갈 수 있다.

### C. 요청 생명주기 설명력 — 15점

- 0점: request가 어떻게 흘러가는지 모른다.
- 5점: 대충 흐름은 보이지만 slot, callback, response ownership이 빠져 있다.
- 10점: submit 이후 흐름을 설명할 수 있다.
- 15점: acquire -> set -> submit -> inline/queue -> callback/release 흐름이 모두 읽힌다.

### D. 내부 자료구조 이해도 — 15점

- 0점: 자료구조가 없다.
- 5점: 타입 이름만 있다.
- 10점: RoutePlan, LockPlan, CmdJob, Slot 구조가 왜 필요한지 보인다.
- 15점: 자료구조가 실행 경로와 연결돼서 이해된다.

### E. 팀 경계 / 수정 경계 명확성 — 10점

- 0점: TCP 팀과 우리 팀 경계가 안 보인다.
- 4점: 팀 경계는 말하지만 파일 수준 수정 경계는 없다.
- 7점: 역할 구분은 된다.
- 10점: “우리가 건드리는 파일”과 “TCP 팀이 위에 붙는 경계”가 파일 기준으로 선명하다.

### F. 코드 조각 품질 — 10점

- 0점: 코드 조각이 없거나 읽을 수 없다.
- 4점: 조각은 있으나 왜 중요한지 설명이 약하다.
- 7점: 조각이 흐름 이해에 도움이 된다.
- 10점: 조각을 보면 함수 역할과 분기 기준을 즉시 이해할 수 있다.

### G. 스터디 순서 / 단계 설계 — 10점

- 0점: 순서가 없다.
- 4점: 순서는 있으나 추상적이다.
- 7점: 파일/함수 중심 읽기 순서가 있다.
- 10점: 처음 읽을 계약, 다음 볼 분기, 마지막에 볼 내부 타입까지 단계가 자연스럽다.

### H. 레이아웃 / 가독성 — 5점

- 0점: 읽기 어렵다.
- 2점: 읽히긴 하지만 학습 자료로는 불편하다.
- 4점: 공부 자료로 충분히 안정적이다.
- 5점: 긴 기술 문장과 코드 조각이 함께 있어도 무리 없이 읽힌다.

## 4. 상한 규칙

아래 조건이 하나라도 걸리면 총점 상한을 둔다.

- 실제 파일 경로 4개 미만: 상한 70
- 실제 함수명/타입명 8개 미만: 상한 72
- 코드 조각 4개 미만: 상한 75
- 읽는 순서 4단계 미만: 상한 78
- request 생명주기 설명 부재: 상한 80
- 팀 경계가 파일 수준이 아니라 문장 수준에만 머묾: 상한 84
- submit 이후 실행 경로가 보드에서 끊김: 상한 86
- 자료구조가 실행 경로와 연결되지 않음: 상한 88

## 5. 감점 규칙

- 실제 코드와 다른 설명: -5
- 함수명은 맞지만 어떤 역할인지 틀리게 설명: -3
- 영문/한글 혼용으로 학습 흐름 방해: -1
- 코드 조각이 너무 작거나 잘려서 사실상 못 읽음: -3
- 발표용 문장 비중이 너무 크고 학습 설명이 부족함: -3
- “좋아 보이는 설명”만 있고 line-level 근거가 없음: -2

## 6. 95점 조건

95점 이상은 아래를 모두 만족해야 한다.

- 필수 조건 전부 충족
- 학습 시작 가능성 14점 이상
- 실제 코드 추적 가능성 19점 이상
- 요청 생명주기 설명력 14점 이상
- 내부 자료구조 이해도 14점 이상
- 팀 경계 / 수정 경계 명확성 9점 이상
- 코드 조각 품질 9점 이상
- 스터디 순서 / 단계 설계 9점 이상
- 레이아웃 / 가독성 4점 이상
- 감점 총합 -2 이하

즉 95점은 “예쁘다”가 아니라,  
**실제로 코드 공부가 된다**는 수준이어야 한다.

## 7. 리뷰 출력 형식
