# 수요 코딩회: 미니 DBMS API 서버

## 1. 목적

- 단 하루, AI를 도구 삼아 팀별 프로젝트에 몰입한다.
- 결과물뿐만 아니라 과정도 중요하다. AI가 작성한 코드를 완벽히 내 것으로 소화해 그 원리를 설명할 수 있어야 한다.
- 낯선 기능도 Top-down으로 빠르게 풀어내며 학습 민첩성을 극대화하고, 현업 레벨의 기술을 직접 부딪쳐보며 실무 감각을 깨운다.

## 2. 요구사항

이번 주 과제는 **미니 DBMS - API 서버**를 구현하는 것이다.

필수 요구사항:

- 구현한 API를 통해 **외부 클라이언트에서 DBMS 기능을 사용할 수 있어야 한다.**
- **스레드 풀(Thread Pool)**을 구성하고, 요청이 들어올 때마다 스레드를 할당하여 **SQL 요청을 병렬로 처리**해야 한다.
- **이전 차수에서 구현한 SQL 처리기와 B+ 트리 인덱스를 그대로 활용**하여 내부 DB 엔진을 구성한다.
- 구현 언어는 **C 언어**다.
- **학습**보다는 **구현**이 우선이다. AI 등을 적극적으로 활용하여 **결과물을 만들어내는 것**이 최우선 목표다.
- 다만, 만들어진 결과물 중 **핵심 로직에 대해서는 이해하고 설명할 수 있어야 한다.**
- 코드를 직접 작성하기 어려운 경우 전체 코드를 AI를 통해 생성해도 괜찮다. 다만, **생성된 코드를 반드시 학습하고 그 원리를 이해**해야 한다.
- 추가적으로, **다른 팀과의 차별점을 둘 수 있는 추가 구현 요소**에 대해서도 함께 고민한다.

## 3. 중점 포인트

- 멀티 스레드 동시성 이슈
- **내부 DB 엔진**과 **외부 API 서버** 사이 연결 설계
- API 서버 아키텍처

## 4. 품질 기준

- 단위 테스트를 통해 함수를 검증한다.
- API 서버를 준비하고, 해당 서버에서 기능 테스트를 통해 API가 제대로 동작하는지 확인한다.
- 엣지 케이스를 최대한 고려한다.
- 이력서와 포트폴리오에 포함할 수 있을 만큼 완성도 높은 수준으로 구현한다.

## 5. 결과물 발표

발표는 **목요일 오전**에 진행한다.

발표 방식:

- 각 팀이 발표를 통해 프로젝트를 서로 공유한다.
- 본인 팀에서 만든 프로젝트를 간단히 데모한다.
- 발표자료는 따로 만들지 않고, GitHub repository의 `README.md`를 기준으로 설명한다.
- AI를 활용해 빠르게 구현하는 것만큼, 생성된 코드에 대한 **철저한 검증과 테스트**도 중요하다.
- 반드시 테스트 케이스를 작성하고 검증 과정을 수행한 뒤, 그 내용을 데모 발표에도 포함한다.

시간:

- 발표 시간: 4분
- QnA 시간: 1분
