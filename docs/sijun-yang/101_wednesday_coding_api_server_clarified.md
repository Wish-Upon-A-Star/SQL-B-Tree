# 수요 코딩회: 미니 DBMS API 서버 요구사항 명확화

이 문서는 `100_wednesday_coding_api_server.md`만 기준으로 “API 서버” 요구사항을 더 명확히 나눈다. 실제 코드 구조나 현재 구현체 이름은 사용하지 않는다.

핵심은 과제에서 말하는 서버가 **외부 서비스를 호출하는 클라이언트**가 아니라, **외부 클라이언트가 DBMS 기능을 사용할 수 있도록 요청을 받는 프로그램**이라는 점이다.

## 1. 용어 정리

| 용어 | 의미 |
| --- | --- |
| API 서버 | 우리가 구현하는 결과물. 외부 클라이언트의 요청을 받아 미니 DBMS 기능을 제공하는 서버 프로그램 |
| 외부 클라이언트 | API 서버에 접속해 SQL 실행 같은 DBMS 기능을 요청하는 사용자 프로그램 |
| 내부 DB 엔진 | 이전 차수에서 구현한 SQL 처리기와 B+ 트리 인덱스를 활용해 실제 DB 기능을 수행하는 부분 |
| 요청 처리부 | 클라이언트 요청을 받아 내부 DB 엔진에 전달하고 응답을 만드는 부분 |
| 스레드 풀 | 여러 요청을 병렬로 처리하기 위해 서버 내부에 두는 작업자 스레드 묶음 |

이 문서에서 “서버”는 항상 API 서버를 뜻한다. 외부 클라이언트는 서버가 아니며, 과제 결과물의 사용 주체다.

## 2. 과제 목표 재정의

100번 문서의 요구사항을 풀어 쓰면 목표는 다음과 같다.

```text
외부 클라이언트
        -> API 요청
        -> API 서버
        -> 요청 처리부
        -> 스레드 풀
        -> 내부 DB 엔진
        -> API 응답
        -> 외부 클라이언트
```

즉, 구현해야 하는 것은 “DBMS 기능을 외부에서 사용할 수 있게 하는 서버”다. 서버가 다른 서버에 접속해 데이터를 가져오는 클라이언트 역할을 한다는 뜻이 아니다.

## 3. 책임 경계

### 외부 클라이언트

외부 클라이언트는 API 서버의 사용자다.

- API 서버에 요청을 보낸다.
- SQL 실행이나 DBMS 기능 사용을 요청한다.
- 서버가 돌려준 응답을 해석한다.
- 테스트나 데모를 위해 만들 수 있지만, 과제의 핵심 구현 대상은 아니다.

### API 서버

API 서버는 과제의 핵심 결과물이다.

- 외부 클라이언트가 접근할 수 있는 API를 제공한다.
- 클라이언트 요청을 읽고 기본 검증을 수행한다.
- SQL 요청을 내부 DB 엔진이 처리할 수 있는 작업으로 넘긴다.
- 처리 결과를 외부 클라이언트가 이해할 수 있는 응답으로 돌려준다.
- 여러 클라이언트 또는 여러 요청이 들어오는 상황을 고려한다.

### 요청 처리부와 스레드 풀

요청 처리부는 API 서버와 내부 DB 엔진 사이의 연결 지점이다.

- 들어온 요청을 작업 단위로 만든다.
- 스레드 풀의 작업자 스레드에 요청 처리를 배정한다.
- 동시에 여러 SQL 요청을 처리할 수 있게 한다.
- 요청 처리 중 발생한 성공/실패 결과를 응답으로 정리한다.

100번 문서의 “요청이 들어올 때마다 스레드를 할당”은 요청마다 새 스레드를 무조건 생성한다는 뜻으로 좁게 해석하지 않는 편이 좋다. 스레드 풀을 구성한다는 요구와 함께 보면, 미리 준비된 스레드 풀에 요청 작업을 배정하는 구조가 더 자연스럽다.

### 내부 DB 엔진

내부 DB 엔진은 기존 미니 DBMS 기능을 수행하는 부분이다.

- 이전 차수에서 구현한 SQL 처리기를 재사용한다.
- 이전 차수에서 구현한 B+ 트리 인덱스를 재사용한다.
- SQL 실행 결과를 요청 처리부에 돌려준다.
- API 서버의 통신 방식 자체를 알 필요는 없다.

## 4. 명확해진 필수 요구사항

1. API 서버는 외부 클라이언트가 접근할 수 있는 API를 제공해야 한다.
2. 외부 클라이언트는 API를 통해 미니 DBMS 기능을 사용할 수 있어야 한다.
3. API 서버 내부에는 요청을 병렬 처리하기 위한 스레드 풀이 있어야 한다.
4. SQL 요청은 스레드 풀을 통해 병렬 처리될 수 있어야 한다.
5. 내부 DB 엔진은 이전 차수의 SQL 처리기와 B+ 트리 인덱스를 재사용해야 한다.
6. 구현 언어는 C다.
7. 핵심 로직은 AI를 활용해 만들더라도 설명할 수 있어야 한다.
8. 함수 단위 테스트와 API 수준 기능 테스트를 모두 고려해야 한다.
9. 엣지 케이스와 동시성 문제를 고려해야 한다.

## 5. 아직 결정해야 하는 설계 항목

100번 문서만 보면 다음 항목은 구체적으로 정해져 있지 않다. 따라서 구현 전에 팀에서 명시해야 한다.

| 항목 | 결정해야 하는 내용 |
| --- | --- |
| API 통신 방식 | TCP, HTTP, 파일 기반 요청, CLI wrapper 등 어떤 방식으로 외부 클라이언트가 접근할지 |
| 요청 형식 | SQL 문자열만 받을지, request id나 operation 필드를 둘지 |
| 응답 형식 | 성공/실패, 오류 메시지, 조회 결과를 어떤 구조로 돌려줄지 |
| 병렬 처리 정책 | 요청 큐를 둘지, 스레드 풀 크기를 어떻게 정할지, 요청이 몰릴 때 어떻게 거절할지 |
| DB 동시성 정책 | 동시에 들어온 SQL이 같은 데이터에 접근할 때 어떻게 보호할지 |
| 테스트 방식 | 단위 테스트와 API 기능 테스트를 어떤 입력/출력 기준으로 검증할지 |

예를 들어 TCP를 선택할 수는 있지만, 100번 문서 자체가 TCP를 필수로 지정한 것은 아니다. TCP를 채택한다면 그때 “API 서버는 TCP 연결을 받는 서버”라고 명확히 쓰면 된다.

## 6. 검증 기준

다음이 가능하면 과제 요구사항에 맞는 결과물로 볼 수 있다.

- API 서버를 실행할 수 있다.
- 외부 클라이언트 역할의 테스트 코드나 도구가 API 서버에 요청을 보낼 수 있다.
- SQL 요청이 내부 DB 엔진까지 전달되고 결과가 응답으로 돌아온다.
- 여러 요청을 동시에 보내도 서버가 안정적으로 처리한다.
- 스레드 풀을 통해 요청이 병렬 처리되는 구조를 설명할 수 있다.
- 기존 SQL 처리기와 B+ 트리 인덱스를 어떻게 재사용했는지 설명할 수 있다.
- 오류 입력, 동시 요청, 정상 SQL 요청을 테스트로 검증한다.
